# План создания ноутбука для сравнения моделей

В этом документе будет описана структура того, что должно быть на выходе в model_search.ipynb. 

Основная идея в том, чтобы показать ряд метрик, по которым можно было бы сравнить реальные треки со смоделированными в unity.

## В начале ноутбука

### Параметры симуляции (Unity)
- 51 агент выходят из одной точки
- Интервал старта: 2–6 секунд между агентами
- Количество просмотренных точек: от 5 до 25 на агента
- Loop: выключен
- Избегание толпы: периодически проверяется, не слишком ли много других агентов в радиусе текущей цели (радиус ~2 м). Если да (≥ 3 человека) — эта точка пропускается, агент переходит к следующей в своём списке.

### 2 картинки: треки реальные и треки симулированные
- **tracks_real_floor0.png** — реальные треки из датасета, наложенные на план этажа
- **tracks_simulated_floor0.png** — симулированные треки из Unity, наложенные на план этажа 

Есть 2 уровня: макро и микро уровень. Макро уровень отвечает за перемещение и движение в рамках этажа, а микро уровень - за движение в конкретных зонах. 

## Макро уровень

### Room popularity ranking - популярность каждой зоны

Нужно взять слой с этажами, там есть зоны с 0-15. Задача взять треки по этажу и посчитать как часто туда заходили агенты и выстроить цепочку по популярности от наименее к более популярному. Такой формат: zone | n_agents_visited | rank. 

После этого необходимо сравнить популряность реальную с популряностью на основе симулированных треков. 

Сейчас параметр считается в room_popularity.py. 

### Edge-load - показывает популярность проходов

Сейчас параметр вычисляется в модуле transition_matrix.py. 

На выходе необходимо сравнить популярность переходов реальную с популярностью на основе симулированных треков.

### Гипотиза:

В смоделированных треках и в реальных эти 2 параметра должны иметь похожий паттерн.

### Входные параметры

1. DXF с планом и зонами (план этажа, зоны 0–15)
2. Папка с CSV-треками реальными (normalized_trajectories) — 1 файл = 1 траектория, ~51 траектория этажа 0
3. DXF с симулированными треками (слой Simulation) — тот же файл
4. Названия слоёв: Floor_plan, Area, Simulation

Название слоев по умолчанию: 
Floor_plan - план этажа
Area - зоны 
Simulation - треки из симуляции

Пути по умолчанию: 
PATH_DXF = model_search/Floor_0.dxf (план, зоны, симулированные треки)
PATH_TRAJECTORIES_CSV = bird-dataset-main/data/normalized_trajectories (реальные треки)



## Микро уровень

В начале секции должно быть 2 картинки:

- смоделированные треки 
- реальные треки

### Density 

Параметр показывает плотность посетителей на единицу площади. Размер сетки по умолчанию 1м*1м и должен быть вынесен во входные параметры.

Сейчас плотность для реальных треков считается в модуле density.py, а для смоделированных - density_Unity.py

На выходе нужно получить карты плотности по реальным и симуляции.

### Time of Presense - время присутствия

Время присутствия = Время выхода - время захода. Это накопительный параметр по ячейке - считаеся суммарно простояли в ячейке в сумме секунд все агенты/люди.

Сейчас параметр для реальных треков считается в модуле density.py, а для смоделированных - density_Unity.py

На выходе нужно получить карты ToP по реальным и симуляции.

### Stop duration distribution

Записываем все остановки по всем агентам и считаем по полученной последовательности некоторую статистику.

Сейчас параметр для реальных треков считается в модуле density.py, а для смоделированных - density_Unity.py

На выходе нужно по смоделированным и по реальным:
— mean value
— median
— 75th and 90th percentiles
— proportion of long stops

## Требование для ноутбука 

В ноутбук не надо пихать горы кода. Нужно код хранить в отдельных модулях. Ноутбук будет использоваться быстрый способ посмотреть метрики и графики, а не для просмотра кучи кода.
